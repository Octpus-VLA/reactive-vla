# create-pr
 
現在のブランチの差分とコミット履歴を読み取り、変更の意図を汲んだ **Conventional Commits 形式のタイトル**と**定型フォーマットの本文**を持つ Pull Request を `gh pr create` で作成するスキル。
 
PR 作成は「差分を読む → 意図を言語化する → タイトル規約に合わせる → 本文を埋める → push して作成する」という毎回ほぼ同じ手順の繰り返しで、地味に時間がかかる。この定型作業を肩代わりするのが狙い。複数のリポジトリで使い回せるよう、ベースブランチやスコープはハードコードせずリポジトリごとに判定する。
 
## いつ使うか
 
ユーザーが明示的に PR 作成を依頼したときだけ発動する。例:
 
- 「PR作って」「プルリク作成して」「この変更でPR出して」
- `/create-pr`、`/create-pr @reviewer`、`/create-pr --draft`
逆に、単にコミットしたい・差分を見たいだけのときは発動しない。
 
## 必要なツール・前提
 
- **git** … 差分・履歴・ブランチ状態の取得。
- **gh CLI（認証済み）** … PR の作成。`gh auth status` で確認できる。未認証なら、その場で `gh auth login` をユーザー自身に実行してもらうよう促す（認証はユーザー権限が必要なので肩代わりしない）。
- 実行にはネットワークとユーザー権限が必要。**制限された sandbox では実行せず**、権限のある環境で動かすこと。
## 実行フロー
 
以下を順に進める。各コマンドは結果を読み、状況に応じて分岐する。
 
### 1. 事前チェック
 
```bash
git rev-parse --is-inside-work-tree   # gitリポジトリか
gh auth status                         # gh認証済みか
git branch --show-current              # 現在のブランチ
```
 
- git リポジトリでない／gh 未認証なら、その旨を伝えて中断する。
- **現在のブランチがベースブランチそのもの（main / master / develop など）の場合は警告する。** 通常 PR はトピックブランチから作る。そのまま進めるかユーザーに確認する。
### 2. ベースブランチの自動判定
 
ハードコードせず、リポジトリの実態から決める。**基本はリモートに設定されたデフォルトブランチ（多くは `main`）をそのまま第一候補にする。**
 
```bash
# リモートのデフォルトブランチを取得（これが第一候補）
gh repo view --json defaultBranchRef --jq .defaultBranchRef.name
# gh が使えない場合のフォールバック
git symbolic-ref refs/remotes/origin/HEAD --short 2>/dev/null | sed 's@^origin/@@'
 
# 参考: GitFlow 系リポジトリかどうかの判定材料として develop の有無も見る
git ls-remote --heads origin develop
```
 
判定ロジック: ベース候補は**リモートのデフォルトブランチ**（GitHub のリポジトリ設定で指定されているブランチ。多くは `main`）。`develop` を自動で優先しない。ただし `origin/develop` が存在する GitFlow 系リポジトリでは、デフォルトブランチではなく `develop` を狙っていることもあるため、**develop が存在する場合のみプレビューで「develop に向けることもできる」と一言添える**（自動では切り替えない）。
 
**判定したベースは後段のプレビューで必ず提示し、ユーザーが訂正できるようにする**（自動判定はあくまで初期値）。
 
### 3. 変更内容の分析
 
```bash
git status                       # 未コミットの変更があるか
git diff <base>...HEAD           # ベースとの差分（変更の中身）
git log <base>..HEAD --oneline   # このブランチで積んだコミット
```
 
- **未コミットの変更がある場合は、勝手にコミットしない。** 内容を要約してユーザーに見せ、「これらをコミットしてから PR に含めますか？」と確認する。承認されたら、差分から適切なコミットメッセージ（同じく Conventional Commits 形式）を提案してコミットする。
- `<base>..HEAD` にコミットが 1 件も無い場合は、PR にできる変更が無い旨を伝えて中断する。
### 4. タイトルと本文の生成
 
差分とコミット履歴から「何を・なぜ変えたか」を抽出する。
 
**タイトル**は Conventional Commits 形式（`<type>(<scope>): <説明>`）。
 
- **type** は変更の性質から推定する: `feat`（機能追加）/ `fix`（バグ修正）/ `refactor` / `docs` / `test` / `chore` / `perf` / `style` / `ci` / `build`。
- **scope** は変更ファイルのパスから推定する。最も変更が集中しているトップレベルのモジュール／ディレクトリ名を使う（例: 変更が `src/needs/` 配下に集中 → `needs`、`apps/web/` → `web`）。スコープが横断的で絞れない場合は省略する（`feat: ...`）。リポジトリ固有のスコープ一覧をハードコードしないこと。
**本文**は以下のテンプレートに沿って、推測ではなく差分から読み取れた事実で埋める。該当が無いセクションは空欄のままにせず「特になし」と書くか、テスト項目のように妥当なものを補う。
 
```markdown
## 概要
[変更の目的と背景を1-2文で簡潔に]
 
## 変更内容
- [主要な変更点を箇条書き]
- [追加・修正・削除した機能]
 
## 影響範囲
- [影響を受けるモジュール・機能]
- [破壊的変更がある場合は明記]
 
## テスト
- [ ] 手動テスト完了
- [ ] 既存機能への影響確認
- [ ] [その他必要なテスト項目]
 
## 備考
[補足情報、関連Issue、注意事項など]
```
 
変更が大きい場合は「変更内容」を機能・モジュール単位で小見出しに分けて整理する。
 
### 5. プレビューと確認（必須）
 
push と PR 作成は外部に公開される取り消しの難しい操作なので、**実行前に必ず全体像を提示して明示的な承認を得る**。以下をまとめて見せる:
 
- ベースブランチ → ヘッドブランチ（例: `main ← feature/xxx`）と、それが自動判定であること
- 生成したタイトル
- 生成した本文
- 含まれるコミット一覧
- push が必要か（未 push の場合）
- draft かどうか、指定されたレビュワー・ラベル
ユーザーが OK と言うまで次に進まない。修正依頼があればタイトル・本文・ベースを直して再提示する。
 
### 6. push と PR の作成
 
承認が取れてから実行する。
 
```bash
# リモートに未 push なら先に push
git push -u origin HEAD
 
# PR 作成（本文は一時ファイル経由が安全）
gh pr create --base <base> --head <current-branch> --title "<title>" --body-file <body-file>
```
 
- `--draft` 指定時は `--draft` を付ける。
- レビュワー指定時は `--reviewer <user>`、ラベル指定時は `--label <label>` を付ける。
- 本文に改行やマークダウンが多いので、`--body` で直書きせず一時ファイル + `--body-file` を使うと崩れにくい。
- 作成後、`gh pr create` が返す PR の URL をユーザーに伝える。
## 引数の解釈
 
`/create-pr` に続く引数を次のように扱う:
 
- `@username` … その人をレビュワーに追加（`--reviewer`）。
- `--draft` … ドラフト PR として作成。
- ベースブランチ名らしき引数（例: `--base main`）… 自動判定を上書きする。
## 注意事項
 
- **コミットが完了してから PR を作る。** 未コミットの差分があるまま作成しない。
- **ベース／ヘッドが同じ、またはヘッドがベースブランチそのものの場合は警告する。**
- **自動判定したベースブランチは必ず確認に出す。** GitFlow か GitHub-flow かはリポジトリで異なるため、思い込みで進めない。
- リポジトリ固有の値（ベースブランチ名・スコープ一覧）をスキル内にハードコードしない。毎回リポジトリから判定する。
- ユーザー権限が必要なため、制限された sandbox モードでは実行しない。
## 使用例
 
```bash
# 基本
/create-pr
 
# レビュワーを指定
/create-pr @username
 
# ドラフトとして作成
/create-pr --draft
 
# ベースブランチを明示
/create-pr --base main
```
 
## タイトル命名規則（参考）
 
Conventional Commits 形式:
 
- `feat(scope): 機能追加の説明`
- `fix(scope): バグ修正の説明`
- `refactor(scope): リファクタリングの説明`
- `docs(scope): ドキュメント更新の説明`
- `test(scope): テスト追加・修正の説明`
- `chore(scope): その他の変更`
scope は前述のとおりリポジトリの変更パスから推定する（固定リストは持たない）。
